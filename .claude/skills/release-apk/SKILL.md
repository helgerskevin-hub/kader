Kader is an Expo/React Native Android app in `app/`. This skill covers
building the **release APK that goes to GitHub**, not a debug build for
the emulator (see the `run-android` skill for that).

## Why this skill exists

On 2026-07-13 both Thom and Kevin found they could no longer install
updates over their existing installed app ("App niet geïnstalleerd").
The cause was **not** a signing problem — every APK built in this
project, local or released, is signed with the same Expo debug key.

The real cause: `app/android/` is gitignored and generated by
`expo prebuild`. `npx expo run:android` (and a bare `gradlew
assembleRelease`) **skip prebuild if `android/` already exists**, so
the native `versionCode` in `app/android/app/build.gradle` silently
drifts behind `app.json` every time someone bumps the version there
without regenerating the native project. Kevin's 0.1.2 build happened
to come from a fresh prebuild (higher versionCode); every build after
that used the stale, lower versionCode, and Android refuses to install
an APK with a lower versionCode than what's already on the device —
same as a downgrade.

**Never build a release APK with a bare `gradlew assembleRelease` or
`expo run:android`.** Always go through the script below, which forces
a clean prebuild and verifies the result before it can be shipped.

## Prerequisites

Same as `run-android`: JDK 17 in `JAVA_HOME` (not the JBR bundled with
Android Studio), `ANDROID_HOME` set, and the Gradle
auto-provisioning workaround in `~/.gradle/gradle.properties`. See
that skill for the exact commands and gotchas if the build fails with
`NoSuchFieldError ... IBM_SEMERU`.

## Build & verify

1. Decide the new version. In `app/app.json`, bump **both**
   `expo.version` (e.g. `"0.1.4"`) and `expo.android.versionCode`
   (integer, always higher than the last one that could plausibly be
   on someone's phone — check `CHANGELOG.md` and prior GitHub release
   assets if unsure, don't just increment from `app.json`'s previous
   value, since past releases have drifted from it before).
2. Run the release script:
   ```powershell
   $env:JAVA_HOME = "C:\Program Files\Eclipse Adoptium\jdk-17.x.x-hotspot"
   $env:ANDROID_HOME = "$env:LOCALAPPDATA\Android\Sdk"
   Set-Location D:\dev\crypto-market\app
   npm run release:apk
   ```
   This runs `app/scripts/bouw-release.mjs`, which:
   - runs `expo prebuild -p android --clean` (forces a fresh native
     project so `build.gradle` picks up the current `app.json`);
   - runs `gradlew assembleRelease`;
   - reads the built APK back with `aapt2 dump badging` and **fails
     hard** if versionCode/versionName don't match `app.json`;
   - runs `apksigner verify --print-certs` and **fails hard** if the
     signing certificate isn't the known debug-key SHA-256
     (`fac61745dc0903786fb9ede62a962b399f7348f0bb6f899b8332667591033b9c`);
   - on success, copies the APK to `app/kader-<version>.apk`.
3. If the script's own checks pass, the APK is safe to upload. If it
   fails, **do not upload anyway** — read the error, it tells you
   exactly which check (version or signing) didn't match.

## Gotcha: stale gradle daemons lock `android/` on `--clean`

If `android/` was left in a partial or locked state from a previous
build attempt, `expo prebuild --clean` fails with `EBUSY: resource
busy or locked` trying to delete it. This happens when a gradle daemon
(spawned by an earlier build, or by an editor's Gradle extension) still
holds file handles open inside `android/app/build/`.

Fix: stop leftover **gradle build daemons** (`java.exe` processes
running `org.gradle.launcher.daemon.bootstrap.GradleDaemon`, check
their command line with
`Get-CimInstance Win32_Process -Filter "ProcessId=<pid>" | Select CommandLine`
to confirm before killing anything — don't kill an IDE's own long-lived
extension host, only the daemon processes). If `rd /s /q` on
`app\android` still fails afterward via PowerShell's `Remove-Item`,
retry with `cmd /c "rd /s /q ...\app\android"` — it's more tolerant of
transient Windows file locks than `Remove-Item`.

## Signing: do not change it

The debug keystore (`app/android/app/debug.keystore`, regenerated
identically by every `expo prebuild`) is what every installed copy of
Kader is currently signed with. Introducing a real release keystore
would change the signing certificate and force every user to
uninstall the old app before installing the new one (all local data —
portfolio, saved traders — is lost). Don't do this without discussing
it first; it defeats the whole point of this skill.

## Publishing

After the script succeeds:
```bash
gh release create v<version> app/kader-<version>.apk --title "Kader v<version>" --notes "..."
```
Update `CHANGELOG.md` and `app/src/changelog.ts` (Dutch, newest entry
first) before or alongside the release — see the "Version control &
releases" section of the root `CLAUDE.md`.
